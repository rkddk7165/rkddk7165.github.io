TROUBLESHOOTING 01

Hot 게시글 조회 API의
동시성 + 성능 문제

게시글 상세 조회 시 조회수 증가 로직에서 Hot 게시글, 즉 동일 postId에 트래픽이 집중되면 DB write 경합으로 지연이 급증했습니다. Atomic Update와 JPA 비관/낙관 락을 비교했지만, Hot row에서는 락 대기와 충돌이 병목을 더 키운다는 것을 확인했습니다.

결론은 락 튜닝이 아니라 요청당 DB write 제거였습니다. Caffeine 로컬 캐시에 조회수 증가량을 누적하고, 주기적으로 batch flush하는 Write-Behind 구조로 전환했습니다.

View Count Write-Behindrequest write 제거
RequestCaffeine CounterBatch FlushMySQL
64 → 453~483 req/s처리량 약 6.0~7.6배 증가
2.27s → 0.45~0.59sp95 지연 약 74~80% 개선

정합성만 보는 문제가 아니라 정합성과 성능을 동시에 만족해야 하는 문제로 정의했고, k6 VU 최대 200 시나리오로 성공률, 처리량, p95를 비교했습니다.

TROUBLESHOOTING 02

게시글 검색 API
LIKE → FullText → OpenSearch

기존 검색은 title/content LIKE '%keyword%' 조건으로 post 테이블 풀스캔과 대량 정렬이 발생했습니다. 2만 건 중 약 6천 건이 매칭되는 조건에서 최신순 21건만 필요했지만, 후보 생성과 정렬 비용이 크게 발생했습니다.

먼저 검색 후보 postId만 조회하고 상세 조립을 분리하는 2-Phase 구조로 병목을 분해했습니다. 이후 MySQL FullText와 OpenSearch PoC를 비교해 검색 워크로드가 커질 때의 개선 방향과 운영 트레이드오프를 검토했습니다.

LIKE~200ms풀스캔 + 대량 정렬
FullText~138ms후보 탐색 개선, 최신순 한계 존재
OpenSearch11~30ms후보 탐색 약 78~92% 감소
도입 전 검토한 트레이드오프

MySQL과 OpenSearch 간 데이터 이중화, 인덱싱/동기화 파이프라인, 장애 시 fallback, 운영 비용을 함께 고려해 바로 도입이 아니라 PoC 기반 조건부 도입 전략을 선택했습니다.
