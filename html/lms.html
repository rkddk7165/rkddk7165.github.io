학사정보시스템

수강신청, 수강 과목 관리 등 학사 행정 기능을 제공하는 시스템입니다. 실제 수강신청 시간대의 트래픽 폭주를 가정해 강의 목록 조회 지연을 k6로 재현하고, DB 접근 패턴과 JPA N+1을 중심으로 개선했습니다.

2025.05 — 2025.07

GitHub Repository
TROUBLESHOOTING

기능 정상 동작을 넘어 실사용 트래픽을 기준으로 봤습니다

500명 동시 접속, 100개 과목 데이터 기준으로 k6 부하 테스트를 수행해 평균 2,286ms, 최대 7,920ms의 응답 지연을 확인했습니다. 이후 DB 접근 횟수와 조회 쿼리 형태를 기준으로 원인을 분리했습니다.

문제는 기능 오류가 아니라 수강신청 시간대처럼 요청이 몰리는 조건에서 사용자 경험을 해치는 성능 병목이었습니다. 따라서 단순 코드 수정이 아니라 동일 부하 조건에서 재측정 가능한 개선 과제로 정의했습니다.

N+1
JPA Lazy Loading

Professor, Syllabus 반복 조회 제거

강의 목록 화면에서 실제 필요한 데이터 범위를 기준으로 Fetch Join 대상을 정했습니다.

k6
Load Test

동일 조건 재측정

변경 후 같은 부하 조건으로 다시 측정해 개선 효과를 수치로 검증했습니다.

TROUBLESHOOTING 01

강의 목록 조회 지연을
부하 테스트로 먼저 재현했습니다

수강신청 페이지의 강의 목록 조회는 평상시에는 정상처럼 보였지만, 동시 접속이 늘어나면 평균 응답이 2초를 넘고 최대 응답은 7초대까지 증가했습니다.

k6로 500명 동시 접속, 100개 과목 데이터를 기준으로 병목을 재현한 뒤, 요청 경로에서 DB 쿼리 패턴과 조회 구조를 추적해 병목 원인을 단계적으로 분리했습니다.

500
Load Scenario

500명 동시 접속 조건 구성

실제 수강신청 시간대의 트래픽 폭주를 가정해 강의 목록 조회 API를 반복 호출하는 시나리오를 만들었습니다.

DB
Root Cause

DB 접근 횟수와 쿼리 형태 추적

응답 지연을 애플리케이션 로직과 DB 조회 구조로 나눠 보고, 반복 SELECT와 연관 객체 로딩을 중심으로 원인을 좁혔습니다.

97%
Result

평균 응답 2,286ms → 71ms

동일 조건 재측정에서 평균 응답 시간을 약 97% 줄이고, 대량 트래픽에서 처리량을 2.7배 개선했습니다.

TROUBLESHOOTING 02

JPA 지연 로딩으로 인한
N+1 쿼리를 제거했습니다

강의 목록 1회 조회가 강의 수에 비례한 추가 조회로 확장되고 있었습니다. 각 강의마다 Professor와 Syllabus가 지연 로딩되며 반복 SELECT가 발생했고, 동시 접속 상황에서는 DB 병목으로 이어졌습니다.

Course 조회 시 수강신청 화면에서 실제 필요한 Professor, Syllabus 범위를 기준으로 Fetch Join 대상을 정했습니다. 변경 후 동일 k6 조건으로 재측정해 초 단위 지연이 100ms대까지 줄어드는 것을 검증했습니다.

N+1
Lazy Loading

Professor / Syllabus 반복 조회

쿼리 로그에서 동일 패턴의 SELECT가 반복되는 것을 확인하고, 강의 목록 조회의 주요 병목으로 확정했습니다.

JOIN
Fetch Join

화면에 필요한 데이터만 한 번에 로딩

무작정 조인 범위를 넓히지 않고, 목록 화면이 실제로 사용하는 데이터 기준으로 연관 데이터를 함께 가져오도록 조정했습니다.

132
Tail Latency

N+1 개선 후 평균 132ms

최대 응답 시간도 1.23초로 줄어 테일 지연이 개선됐고, 수강신청 시나리오에서 체감 지연을 낮췄습니다.
